# Описание микрофронтендной архитектуры Pizza App

## Общая концепция

Приложение Pizza App реализовано в двух вариантах: классический монолитный SPA и микрофронтендная версия. Оба варианта работают параллельно на одном сервере и используют общий бэкенд. Монолит доступен на домене `pizza.ew-production.ru`, микрофронтендная версия — на `micro.pizza.ew-production.ru`. Такой подход позволяет наглядно сравнить две архитектуры на одном и том же функционале.

---

## Способ интеграции: Module Federation

Для интеграции микрофронтендов выбран подход **Module Federation** — механизм, позволяющий нескольким независимым JavaScript-приложениям в рантайме обмениваться модулями через сеть. В качестве реализации используется библиотека `@module-federation/vite` версии 1.9.3, адаптированная для сборщика Vite.

Каждый микрофронтенд является самостоятельным Vite-приложением, которое при сборке генерирует файл `remoteEntry.js` — манифест, описывающий экспортируемые модули и их зависимости. Хост-приложение (Shell) на этапе рантайма загружает эти манифесты с удалённых серверов и динамически подключает нужные модули.

Выбор Module Federation обусловлен тем, что этот подход обеспечивает рантайм-интеграцию без необходимости пересборки всего приложения при изменении одного модуля, позволяет дедуплицировать общие зависимости и поддерживает ленивую загрузку модулей по требованию.

## Модули системы

Система состоит из шести независимых модулей:

**Shell (Host-приложение)** — контейнер, который управляет навигацией, отображает общий хедер и динамически загружает микрофронтенды в зависимости от текущего маршрута. Shell не содержит бизнес-логики — он выполняет роль оркестратора.

**Auth** — модуль аутентификации. Отвечает за форму входа, проверку токена и управление сессией пользователя. Не имеет внутреннего роутера, представлен одним экраном авторизации.

**Pizza-Builder** — конструктор пиццы. Позволяет выбрать тесто, размер, соус и ингредиенты, сформировать пиццу и добавить её в корзину. Загружает справочные данные (ингредиенты, размеры, соусы, тесто) с бэкенда. Не имеет внутреннего роутера.

**Cart** — корзина. Отображает добавленные пиццы, позволяет изменять количество, добавлять дополнительные товары и оформлять заказ. Имеет собственный внутренний роутер для управления навигацией внутри модуля.

**Profile** — профиль пользователя. Управление личными данными и адресами доставки. Имеет собственный внутренний роутер.

**Order** — история заказов. Отображение списка оформленных заказов с возможностью удаления. Не имеет внутреннего роутера.

## Общие зависимости

Общие зависимости разделяются на два уровня.

На уровне Module Federation через механизм `shared` с параметром `singleton: true` разделяются фреймворк Vue 3 и библиотека управления состоянием Pinia. Параметр `singleton` гарантирует, что в браузере загружается ровно одна копия каждой библиотеки, независимо от количества подключённых микрофронтендов. Это исключает дублирование кода и обеспечивает корректную работу внутренних механизмов Vue, таких как dependency injection. Маршрутизатор Vue Router намеренно не включён в shared-зависимости: каждый микрофронтенд, которому необходим роутер, создаёт собственный экземпляр.

На уровне npm-пакетов общие UI-компоненты и типы выделены в пакет `@pizza/shared`, который публикуется в локальный npm-реестр Verdaccio. Пакет содержит переиспользуемые компоненты интерфейса (кнопки, поля ввода, счётчики, выпадающие списки, боковые панели), TypeScript-типы и справочные данные. Каждый микрофронтенд устанавливает этот пакет как обычную npm-зависимость, что обеспечивает единообразие интерфейса и типовую безопасность во всех модулях.

## Коммуникация между модулями

Для коммуникации между микрофронтендами выбран подход на основе пользовательских событий браузера (Custom Events), диспатчимых на объекте `window`. Этот подход обеспечивает слабую связанность модулей: отправитель события не знает о получателях, а получатель не зависит от реализации отправителя.

В системе используются следующие события:

- `auth:login-success` — генерируется модулем Auth после успешной авторизации. Shell перехватывает это событие и перенаправляет пользователя на главную страницу.
- `auth:close` — генерируется при выходе из системы. Shell выполняет редирект.
- `pizza:add-to-cart` — генерируется модулем Pizza-Builder при добавлении пиццы. Передаёт полный объект пиццы (название, тесто, размер, соус, начинки, цена). Слушатель в модуле Cart сохраняет пиццу в localStorage.
- `cart:total-updated` — генерируется модулем Cart при изменении содержимого корзины. Shell обновляет счётчик суммы в хедере.
- `pizza:go-to-cart` — генерируется для навигации из конструктора в корзину.
- `cart:order-complete` — генерируется после успешного оформления заказа.

Для персистентного хранения данных корзины между модулями используется localStorage с ключом `pizza_cart`. Токен аутентификации хранится через утилиту TokenStorage, также основанную на localStorage. Такой подход позволяет данным пережить перезагрузку страницы и быть доступными всем модулям без прямых зависимостей.

## Стратегия загрузки модулей

Загрузка модулей разделена на два этапа: немедленную (eager) и отложенную (lazy).

При запуске Shell-приложения немедленно загружаются два модуля. Модуль Auth загружается для проверки валидности токена авторизации — это необходимо для отображения имени пользователя в хедере и определения состояния сессии. Из модуля Cart загружается только слушатель событий, который перехватывает событие `pizza:add-to-cart` и сохраняет пиццу в localStorage. Без немедленной загрузки этого слушателя пиццы, добавленные до первого перехода в корзину, терялись бы.

Остальные модули загружаются отложенно, только при навигации пользователя на соответствующий маршрут. При переходе на главную страницу загружается Pizza-Builder, при переходе в профиль — Profile, при переходе к истории заказов — Order, при переходе в корзину — полный интерфейс Cart. Такой подход сокращает время первоначальной загрузки приложения: пользователь получает рабочий интерфейс, не дожидаясь загрузки всех модулей.

## Shell-приложение и маршрутизация

Shell реализует двухуровневую маршрутизацию. На верхнем уровне Shell Router использует Vue Router с режимом `createWebHistory` и управляет URL-адресом в браузере. Все маршруты ведут на единый компонент MfeView, но с различными метаданными, указывающими имя целевого микрофронтенда.

Компонент MfeView извлекает имя микрофронтенда из метаданных маршрута и передаёт его компоненту MfeLoader. MfeLoader выполняет основную работу: через Module Federation Runtime загружает remote-модуль, получает функции `mount` и `unmount`, создаёт DOM-контейнер и вызывает `mount`, передавая в него HTML-элемент. При смене маршрута MfeLoader сначала вызывает `unmount` для текущего модуля, очищая Vue-инстанс и освобождая ресурсы, затем загружает и монтирует новый модуль.

На нижнем уровне некоторые микрофронтенды (Cart, Profile) имеют собственные внутренние роутеры для управления навигацией внутри модуля.

Shell также содержит общий хедер с логотипом, счётчиком корзины и информацией о пользователе. Хедер обновляется через события: `cart:total-updated` для суммы корзины, `auth:login-success` для данных пользователя. На странице авторизации хедер скрывается через флаг `noHeader` в метаданных маршрута.

## Жизненный цикл микрофронтенда

Каждый микрофронтенд экспортирует стандартный контракт из файла `entry.ts`: функцию `mount`, принимающую DOM-элемент, и функцию `unmount` для очистки. Некоторые модули дополнительно экспортируют функцию `init` для инициализации без монтирования интерфейса.

При вызове `mount` микрофронтенд создаёт полностью независимый экземпляр Vue-приложения через `createApp`, подключает собственный экземпляр Pinia для управления состоянием, при необходимости подключает роутер, и монтирует приложение в переданный DOM-контейнер. При вызове `unmount` приложение размонтируется, все ресурсы освобождаются.

Такой подход обеспечивает полную изоляцию модулей: каждый микрофронтенд имеет собственное состояние, собственный жизненный цикл и может быть развёрнут независимо от остальных.

## Деплой

Приложение разворачивается через Docker Compose. Каждый микрофронтенд собирается в отдельный Docker-контейнер с Nginx, раздающим статические файлы. Перед всеми контейнерами стоит Traefik — реверс-прокси, который маршрутизирует трафик по доменам и автоматически выпускает SSL-сертификаты через Let's Encrypt.

Каждый микрофронтенд размещается на собственном субдомене: `auth.pizza.ew-production.ru`, `builder.pizza.ew-production.ru`, `cart.pizza.ew-production.ru`, `profile.pizza.ew-production.ru`, `order.pizza.ew-production.ru`. Shell-приложение размещается на `micro.pizza.ew-production.ru` и при загрузке обращается к субдоменам микрофронтендов за их `remoteEntry.js`. Бэкенд доступен на `api.pizza.ew-production.ru`.

## Технологический стек

- **Фреймворк:** Vue 3.3.8 (Composition API)
- **Управление состоянием:** Pinia 2.1.7
- **Маршрутизация:** Vue Router 4.2.5
- **Сборщик:** Vite 6.2.5
- **Module Federation:** @module-federation/vite 1.9.3
- **HTTP-клиент:** Axios
- **Язык:** TypeScript
- **Стили:** SCSS
- **Бэкенд:** NestJS, PostgreSQL
- **Контейнеризация:** Docker, Docker Compose
- **Реверс-прокси:** Traefik 2.11
- **NPM-реестр:** Verdaccio (для пакета @pizza/shared)
